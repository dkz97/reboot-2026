# Vibe Coding 模板

---

## A. 全局规则（写一次，放到工具的长期指令位置）

> **存放位置：**
> - ChatGPT → Custom Instructions 或 Project Instructions
> - Claude → Project Instructions；Claude Code → CLAUDE.md
> - Cursor → Project/User/Team Rules 或 AGENTS.md

1. 优先最小可行实现，不先做大而全架构。
2. 不要擅自扩展需求；严格以"本轮只做什么 / 不做什么"为边界。
3. 每次只推进一个可验收的小闭环。
4. 先说明准备怎么做、会改哪些文件、主要风险，再开始实现。
5. 完成后必须给出运行方法、验证方法、已知限制。
6. 每个稳定状态都留回退点，再进入下一步。
7. 连续两轮验收不通过时，停止叠改。把当前状态完整整理给我，判断这是实现问题还是方向问题，并建议：继续修、回退到上一个稳定点、还是重拆任务。在给出判断前不继续加新需求。

---

## B. 项目启动卡（每个新项目填一次）

```
# 项目启动卡

## 一句话目标
[做什么 / 给谁用 / 解决什么问题]

## 现有资源
[已有代码 / 数据 / API / 密钥条件 / 参考项目 / 截图]

## 本轮只做什么
[本轮范围]

## 本轮不做什么
[明确排除项]

## 验收标准
- [做到什么算完成]
- [怎么手动验证]

## 约束条件
[技术偏好 / 时间预期 / 可接受简化 / 运行环境]
```

---

## C. 扩展信息块（有就加，没有就跳过）

```
## 外观参考
[截图 / 草图 / 参考链接 / 交互示意]

## 外部依赖
[API 文档 / 鉴权方式 / 数据格式样例 / 第三方限制]

## 输入输出样例
[输入示例 / 期望输出示例 / 正确结果长什么样]
```

---

## D. AI 输出要求（贴在启动卡末尾）

```
请基于以上信息输出：
1. 最小可行实现方案
2. 3~5 个开发步骤
3. 每一步的产出与验收方式
4. 本轮风险点
5. 建议先做的第一步及原因
```

---

## E. 每轮推进卡（每推进一步复制一次）

```
现在做到哪了：
[一句话]

这轮只做：
[本轮目标]

不做：
[边界]

验收：
- [标准1]
- [标准2]

补充上下文：
[相关文件 / 报错 / 截图 / 接口信息 / 运行结果]

请先告诉我：
1. 这一步怎么做
2. 会改哪些文件
3. 风险点是什么
然后再开始实现。
完成后给我运行和验证方法。
```

## F. 止损时整理模板（连续两轮验收不通过时用）

```
当前状态：

想做什么：
[原始目标]

现在做到哪了：
[当前进度]

最近两轮改了什么：
[改动摘要]

报错 / 异常现象：
[具体信息]

主流程是否还能跑：
[能 / 不能]

请判断：
1. 这是实现问题还是方向问题
2. 应该继续修、回退到上一个稳定点、还是重拆任务
```
